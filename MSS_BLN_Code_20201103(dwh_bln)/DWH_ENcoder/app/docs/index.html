EventLink Multi-format Encoder Node

EventLink Multi-format Encoder Node is used for encoding event records in Comptel EventLink internal format into ASCII or binary format for target systems, for example, billing or fraud.

Functionality of EventLink Multi-format Encoder Node

EventLink Multi-format Encoder Node is a generic ASCII and binary data encoder for Comptel EventLink. EventLink Multi-format Encoder Node is responsible for receiving input event records (ERs), formatting the ERs as required by the target systems and forwarding the data to the next node in the process stream for distributing the data to the target system.

EventLink Multi-format Encoder Node uses a data format description to determine the structure and layout of the output data. When EventLink Multi-format Encoder Node receives an input record, it searches for the corresponding output record specification from the data format description and writes the record out accordingly. EventLink Multi-format Encoder Node does not create the output file but forms a data block of the file contents and sends it to the next node together with the desired output file name. It is up to the distributor node to decide how and when the data is distributed or stored.

EventLink Multi-format Encoder Node scheduling

EventLink Multi-format Encoder Node can be configured to close output files when its schedule is run. In the scheduled file closing, the output files are closed unless they are newer than the age defined with the MinAgeOfFile node parameter. Empty output files are created only if the CreateEmptyOutputFiles parameter is set on. EventLink Multi-format Encoder Node can also be configured to reset file sequence numbers in output file names when its schedule is run.

An example of output file name creation is presented below. In this example the original input file name is ‘A031126220638AMA’. The InputFileNameExtractionMask and OutputFileNameCreationMask parameters have the values given as the examples of these configuration parameters.

InputFileNameExtractionMask ‘([A-Za-z]+)([0-9]+)([A-Za-z]+)’ is a regular expression that captures pieces of the original file name (subpatterns are defined with parenthesis). In this case regular expression matches the file name in three parts and stores the match results into variables that can be used to form the final file name. The results are stored into $n variable, where n is the nth match. A maximum of ten enclosed regular expressions are allowed (for more information, see ‘man regexec’ and ‘man regexp(5)’).

OutputFileNameCreationMask ‘$2.&FSIZE.&CREATIONTIME(%d%m%Y).&SEQ(4)’ defines the output file name format. $2 is the second captured match of the regular expression run against the original file name, which in this case it would be the digit string ‘031126220638’. The next item is ‘.’ character followed by the file size in bytes. The next ‘.’ character is followed by a timestamp in DDMMYYYY format. After the last ‘.’ character there is a running sequence number that consists of four (4) digits, thus the sequence number will be between 0001 and 9999. As a result, the following output file name is created (assuming that this is the 24th time an output file name is requested):

031126220638.4096.23032004.0024

Everything that cannot be interpreted as a literal, regular expression capture match ($1,$2,$3,...) or a date specific syntax is inserted into the file name as such.

Regular expression syntax

(Copyright 1991 The Regents of the University of California. All rights reserved.)

A regular expression is zero or more branches, separated by ’|’. It matches anything that matches one of the branches.

A branch is zero or more pieces, concatenated. It matches a match for the first, followed by a match for the second, etc.

A piece is an atom possibly followed by ’*’,’+’, or ’?’. An atom followed by ’*’ matches a sequence of 0 or more matches of the atom. An atom followed by ’+’ matches a sequence of 1 or more matches of the atom. An atom followed by ’?’ matches a match of the atom, or the null string.

An atom is a regular expression in parentheses (matching a match for the regular expression), a range (see below), ’. ’ (matching any single character), ’^’ (matching the null string at the beginning of the input string), ’$’ (matching the null string at the end of the input string), a ’\’ followed by a single character (matching that character), or a single character with no other significance (matching that character).

A range is a sequence of characters enclosed in ’[]’. It normally matches any single character from the sequence. If the sequence begins with ’^’, it matches any single character not from the rest of the sequence. If two characters in the sequence are separated by ’-’, this is shorthand for the full list of ASCII characters between them (for example, ’[0-9]’ matches any decimal digit). To include a literal ’]’ in the sequence, make it the first character (following a possible ’^’). To include a literal ’-’, make it the first or the last character.

Description of variables for output file separation

The input records can be separated to different output files according to the content of an input field or a metadata field. The separated output files will have individual counter management, except the file sequence counters, and the files are closed independently from each other. The output description is common for all output files.

The SeparationCondition parameter can have as a value the following metadata fields that separate input records to output files:

#input_id
#output_id
#input_type
#output_type
#addkey
#source_id
#filename

Any other string value for the SeparationCondition parameter is interpreted as the name of an input field. The values of that input field are used to separate the input records to output files.

If the value for the SeparationCondition parameter is changed while EventLink Multi-format Encoder Node is processing data or it is stopped while it has unfinished output files in internal storage, the unfinished output files are flushed before the new value for the SeparationCondition parameter is taken into use.

If the SeparationCondition parameter has an indefinite number of values, EventLink Multi-format Encoder Node eventually exhausts all the resources in the executing host. The resources reserved by separated output files can be freed by using the RemoveFileTemplates and RemoveFileTemplatesTimeout parameters.

Writing records to different output links

Input records can be separated to different output links according to configuration XML provided in the TargetLinksConfigurationFileName parameter.

Following is an example configuration file:

<outlinks>

  <outlink name="MFE_OUT_1">

    <keys>

      <key>SDK_SOURCE_1</key>

      <key>SDK_SOURCE_2</key>

      <key>SDK_A*</key>

    </keys>

  </outlink>

  <outlink name="MFE_OUT_2">

    <keys>

      <key>SDK_SOURCE_3</key>

      <key>SDK_SOURCE_4</key>

      <key></key>

    </keys>

  </outlink>

</outlinks>

 

The example configuration file has two output links - MFE_OUT_1 and MFE_OUT_2. A record's SeparationCondition parameter value will be used for matching output link in this configuration.

For example, if the value is SDK_SOURCE_1 or SDK_SOURCE_2, the record will be written to link MFE_OUT_1. If the value is SDK_SOURCE_3 or SDK_SOURCE_4, then the record will be written to link MFE_OUT_2. If the value is empty, it will match the rule "<key></key>", and the record will be written to link MFE_OUT_2.

A key can allow one wildcard character (*) placed at the last position of the key. No multiple wildcards allowed, and the position of the wildcard must be the last. For example, if the key pattern is <key>SDK_A*</key>, values like SDK_A, SDK_A201511201533, and SDK_APLUS will match, and the records will be written to the associated output link (that is MFE_OUT_1 based on the above example configuration).

If a value has no matches, the record will be written to the link defined in the DefaultOutputFilesLink parameter.
